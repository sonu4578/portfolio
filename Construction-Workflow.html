Construction Workflow Platform

An event-driven platform that captures every field-level deviation on construction Material Requisition forms in real time, powering analytics dashboards and immutable audit trails.

Project Overview

The Construction Workflow Platform is built on an event-driven architecture designed for the construction industry. Field workers fill out Material Requisition forms that are pre-populated with system-generated values drawn from the Bill of Materials, vendor rate cards, and project schedules. Every time a worker deviates from a system recommendation — changing a quantity, swapping a vendor, or adjusting a price — the change is captured as an immutable event and made instantly available for real-time visibility and historical analysis.

The platform seamlessly switches between a local developer environment (Kafka + kustainer) and a fully cloud-hosted production environment (Azure Event Hubs + Azure Data Explorer) using only environment variable changes, with zero code modifications required.

Key Features

  • Material Requisition forms pre-filled with BOM quantities, rate-card prices, and preferred vendors
  • Real-time live event feed updated instantly via Server-Sent Events whenever a field is changed
  • Immutable event log stored in both PostgreSQL and Azure Data Explorer
  • Analytics dashboard powered by KQL queries showing material substitution patterns, price deviations, and vendor swap frequency
  • Event streaming via Kafka (locally) and Azure Event Hubs (production) with no code changes required to switch environments

High-Level Architecture


The platform is designed in four layers:

Users & Clients — Field workers and project managers access the platform through a web browser. Field workers submit Material Requisition forms on-site while managers monitor the analytics dashboard.

UI Layer — Built with React and TypeScript. React Hook Form with Zod handles validation. TanStack Query manages data fetching and caching. Server-Sent Events push real-time notifications back to the browser to update the live event feed instantly.

API Layer — Built with NestJS. In a single operation it saves the event to PostgreSQL, updates the form state, publishes to Kafka, writes to Azure Data Explorer, and pushes an SSE notification to the browser.

Data & Event Layer — PostgreSQL stores the current form state and full event history. Kafka / Azure Event Hubs act as the streaming backbone. Azure Data Explorer stores the immutable event log and powers the analytics dashboard via KQL.

Challenges & Solutions

One of the core challenges was designing a system that could run identically on a developer laptop and in a cloud production environment without requiring code changes. This was solved by abstracting the event-streaming and analytics-ingest layers behind environment variables — switching from Kafka to Azure Event Hubs and from kustainer to Azure Data Explorer requires only changing KUSTO_INGEST_MODE and connection strings.

Another challenge was keeping Docker Desktop memory usage manageable while running six infrastructure services simultaneously (Kafka, PostgreSQL, Redis, kustainer, Azurite, Web PubSub). This was addressed by carefully tuning per-container memory limits in docker-compose.yml, keeping total infra usage under ~3 GB.

Delivering a truly real-time event feed to the browser without resorting to polling required implementing Server-Sent Events through NestJS, which pushes change notifications to the frontend the moment an event is persisted — giving field workers and managers an instant, live view of every form deviation.

Results

The platform successfully delivers an end-to-end event-driven workflow where every field-level change made by a construction worker is captured, streamed, stored immutably, and made available for analytics — all in real time. The analytics dashboard surfaces actionable patterns such as which materials are most frequently substituted, which vendors are swapped, and where price deviations are concentrated, enabling project managers to make data-driven procurement decisions.

The environment-parity design means the same monorepo codebase runs in local development and in a fully Azure-hosted production environment with no code modifications, significantly reducing deployment risk and accelerating the path from development to production.
